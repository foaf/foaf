foaf syntactic schemas

Explanation from Leigh Dodds follows:

This package shows the specification (and implementation with Schematron and XSLT) of three syntactic profiles, ie. document formats, based on FOAF RDF/XML.

foaf-image.sch
Simple, restricted syntax for a format for publishing lists of images. Images must have title and creator, and must depict at least one foaf:Person. foaf:Person elements restricted to seeAlso and mbox_sha1sum.

...
foaf-minimal.sch
...

The package includes the following files:

build.xml
Ant build script to generate FOAF-Basic and FOAF-Minimal validators; includes schematron compilation step
schema.rdf
copy of FOAF Schema
/schema
basic schemas
/xsl
stylesheets to generate Schematron assertions based on poking around in the schema. E.g. all literals, all properties of foaf:Agent/foaf:Person, etc.
/out
final versions of the schema incorporating data from transforms.
profiles.txt
notes on profiles
ldodds.rdf
test file, with some deliberately broken bits.

We also have Valid and invalid test documents. Notice that foaf:minimal.rdf also validates as foaf-basic.rdf as you'd expect.

just run "ant" to build them both. If you want to make manual changes change the versions under /schema.

Notes from Leigh

This is probably overkill, but it saved me manually typing in a bunch of property names. Should be possible to re-generate them if the schema changes as well, so thats an additional benefit. Would be easy to adapt this to only include properties with a particular version label as well, e.g. FOAF-Minimal with only stable properties. Further parameterising the stylesheets would also allow this to be generalised to other vocabularies, so there may be some additional mileage.

foaf-basic and foaf-minimal schemas are probably complete enough for demoing purposes.

They may not be as strict as they might be, but without clearer specification of the syntax constraints it's hard to know where to go next. Happy to add more constraints if you think of 'em, or based on discussions. See schema comments for a list of implemented constraints.

Hopefully this meets your needs. I can also look at a RELAX NG versions of the schemas (in fact I think that they could also be done with DTDs, no really complicated constraints) if you think that's necessary, but the Schematron versions make better online validator tools methinks.

This has been an interesting exercise. Makes me wonder whether it would be possible to generalise the stylesheets I created to grab lists of properties from the schema into a more general framework. If syntactic profiles become common-place (and I'm beginning to wonder whether this subset/fork/syntax debate is going to recur for any popular RDF format!) then the may also be mileage in defining a way to declaratively say, "these classes and properties are valid in this profile". Based on my tinkerings here, this looks possible.

Profile Descriptions (draft)

There are various kinds of RDF/XML document that use FOAF. Here we try to characterise some particular use cases, so that syntactic schemas, test data etc. can be produced.

FOAF-Minimal

Minimally useful FOAF. Basically just a series of properties about a person. Should allow only a single foaf:Person, any properties associated with foaf:Person or foaf:Agent, no elements from other namespaces, no other foaf elements, including foaf:knows.

Sort of like vCard but using FOAF.

FOAF-Basic

"FOAF-a-Matic FOAF". Above, but relaxes the restrictions on foaf:knows, allowing other people to be included, as nested elements only. Again no elements from other namespaces, no other foaf elements.

Lot of FOAF out there falls in this category (intentionally or by accident).

FOAF-Classic

FOAF is all its glorious variability. Any RDF/XML document using FOAF (eg. annotated RSS feeds) would also count.

FOAF-Image

Lists of Images, depictions

FOAF-Biblio

Lists of Documents, makers
